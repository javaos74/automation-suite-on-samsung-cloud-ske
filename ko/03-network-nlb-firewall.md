# 3단계: 네트워크, NLB 및 방화벽 구성

UiPath Automation Suite를 위한 삼성 클라우드 VPC 네트워킹, NLB(네트워크 로드 밸런서) 설정, 방화벽 규칙 구성 가이드입니다.

## 아키텍처 개요

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         삼성 클라우드 VPC                                │
│                                                                         │
│  ┌─────────────┐    ┌──────────────────┐    ┌───────────────────────┐  │
│  │  인터넷     │    │  퍼블릭 서브넷    │    │   프라이빗 서브넷     │  │
│  │  게이트웨이  │───▶│  (로드 밸런서)    │───▶│   (워커 노드)        │  │
│  │             │    │                   │    │                       │  │
│  │  IGW 방화벽 │    │   NLB (L4)       │    │   SKE 워커 노드       │  │
│  │  (인바운드)  │    │   LB 방화벽      │    │   보안 그룹           │  │
│  └─────────────┘    └──────────────────┘    └───────────────────────┘  │
│                                                                         │
│  흐름: 인터넷 → IGW → IGW 방화벽 → 퍼블릭 서브넷 (NLB) → LB 방화벽 →  │
│        프라이빗 서브넷 (워커 + 보안 그룹)                                │
└─────────────────────────────────────────────────────────────────────────┘
```

> **참고:** 삼성 클라우드 VPC 아키텍처의 상세 다이어그램은 `load_balancer_overview.png`를 참조하세요.

## 사전 요구 사항

| 항목 | 요구 사항 |
|------|-----------|
| SKE 클러스터 | 워커 노드가 실행 중인 상태 (2단계) |
| VPC | 퍼블릭 및 프라이빗 서브넷 구성 완료 |
| 퍼블릭 IP | 할당 가능한 상태 |
| DNS | DNS 관리 접근 권한 |

## 환경 변수 설정

```bash
export FQDN="ske.myrobots.co.kr"
export PUBLIC_IP="123.41.32.188"
export PUBLIC_IP_ID="a5168593ce6c44d68e7255606eb07d14"
export NODEPORT_RANGE="30000-32767"
```

## 3-1단계: VPC 아키텍처 이해

삼성 클라우드 VPC는 계층형 방화벽 접근 방식을 사용합니다:

| 계층 | 구성 요소 | 목적 |
|------|-----------|------|
| 1 | 인터넷 게이트웨이 방화벽 | 인터넷에서 들어오는 트래픽 제어 |
| 2 | 로드 밸런서 방화벽 | NLB와 워커 노드 간 트래픽 제어 |
| 3 | 보안 그룹 | 워커 노드 수준의 트래픽 제어 |

### 트래픽 흐름

```
클라이언트 요청 (HTTPS:443)
  │
  ▼
인터넷 게이트웨이
  │ IGW 방화벽: LB Service IP:80,443 인바운드 허용
  ▼
퍼블릭 서브넷 — NLB (로드 밸런서)
  │ LB 방화벽: 워커:NodePort로 아웃바운드 허용
  │            헬스체크/SourceNAT에서 워커:NodePort로 인바운드 허용
  ▼
프라이빗 서브넷 — 워커 노드
  │ 보안 그룹: LB SourceNAT/헬스체크에서 NodePort로 인바운드 허용
  ▼
Istio Ingress Gateway Pod (NodePort 서비스)
  │
  ▼
UiPath 서비스 (VirtualService 라우팅 경유)
```

## 3-2단계: 퍼블릭 IP 생성

1. 삼성 클라우드 콘솔에 로그인
2. **네트워킹 > 퍼블릭 IP** 메뉴로 이동
3. **퍼블릭 IP 생성** 클릭

| 설정 | 값 |
|------|-----|
| 이름 | `uipath-as-public-ip` |
| 리전 | kr-west1 |
| 설명 | UiPath Automation Suite NLB용 퍼블릭 IP |

4. 할당된 IP 주소와 UUID를 기록

### 퍼블릭 IP 확인

```bash
# 해당 값을 기록
echo "퍼블릭 IP: ${PUBLIC_IP}"
echo "퍼블릭 IP UUID: ${PUBLIC_IP_ID}"
```

## 3-3단계: 인터넷 게이트웨이 방화벽 구성

**네트워킹 > 인터넷 게이트웨이 > 방화벽 규칙** 메뉴로 이동

### 인바운드 규칙

| 우선순위 | 방향 | 출발지 | 목적지 | 포트 | 프로토콜 | 동작 |
|----------|------|--------|--------|------|----------|------|
| 100 | 인바운드 | 0.0.0.0/0 (전체) | ${PUBLIC_IP}/32 | 80 | TCP | 허용 |
| 101 | 인바운드 | 0.0.0.0/0 (전체) | ${PUBLIC_IP}/32 | 443 | TCP | 허용 |

> **참고:** 운영 환경에서는 0.0.0.0/0 대신 알려진 클라이언트 IP 범위로 출발지를 제한하세요.

### 확인

```bash
# 외부 네트워크에서 포트 접근 가능 여부 확인 (NLB 생성 후)
nc -zv ${PUBLIC_IP} 443
curl -k https://${PUBLIC_IP} --connect-timeout 5
```

---

## 3-4단계: 로드 밸런서 방화벽 구성

**네트워킹 > 로드 밸런서 > 방화벽 규칙** 메뉴로 이동

### 아웃바운드 규칙 (NLB → 워커 노드)

| 우선순위 | 방향 | 출발지 | 목적지 | 포트 | 프로토콜 | 동작 |
|----------|------|--------|--------|------|----------|------|
| 100 | 아웃바운드 | LB Service IP | 워커 노드 | 30000-32767 | TCP | 허용 |
| 101 | 아웃바운드 | LB Service IP | 워커 노드 | 80 | TCP | 허용 |
| 102 | 아웃바운드 | LB Service IP | 워커 노드 | 443 | TCP | 허용 |

### 인바운드 규칙 (헬스 체크 + Source NAT → 워커 노드)

| 우선순위 | 방향 | 출발지 | 목적지 | 포트 | 프로토콜 | 동작 |
|----------|------|--------|--------|------|----------|------|
| 100 | 인바운드 | 헬스 체크 IP | 워커 노드 | 30000-32767 | TCP | 허용 |
| 101 | 인바운드 | Source NAT IP | 워커 노드 | 30000-32767 | TCP | 허용 |
| 102 | 인바운드 | 전체 | 워커 노드 | 80 | TCP | 허용 |
| 103 | 인바운드 | 전체 | 워커 노드 | 443 | TCP | 허용 |

> **참고:** 헬스 체크 IP와 Source NAT IP는 NLB 생성 시 삼성 클라우드에서 제공됩니다. NLB 상세 페이지에서 해당 값을 확인하세요.

## 3-5단계: 보안 그룹 구성

**네트워킹 > 보안 그룹** (워커 노드 서브넷에 연결됨) 메뉴로 이동

### 인바운드 규칙

| 우선순위 | 방향 | 출발지 | 목적지 | 포트 | 프로토콜 | 동작 |
|----------|------|--------|--------|------|----------|------|
| 100 | 인바운드 | LB Source NAT IP | 워커 노드 | 30000-32767 | TCP | 허용 |
| 101 | 인바운드 | 헬스 체크 IP | 워커 노드 | 30000-32767 | TCP | 허용 |

### 추가 규칙 (노드 간 통신)

| 우선순위 | 방향 | 출발지 | 목적지 | 포트 | 프로토콜 | 동작 |
|----------|------|--------|--------|------|----------|------|
| 200 | 인바운드 | 프라이빗 서브넷 CIDR | 워커 노드 | 전체 | 전체 | 허용 |
| 201 | 아웃바운드 | 워커 노드 | 전체 | 전체 | 전체 | 허용 |

## 3-6단계: DNS 레코드 구성

퍼블릭 IP를 가리키는 DNS 레코드를 생성합니다:

### 필수 DNS 레코드

| 타입 | 이름 | 값 | TTL |
|------|------|-----|-----|
| A | `ske.myrobots.co.kr` | `123.41.32.188` | 300 |
| A | `*.ske.myrobots.co.kr` | `123.41.32.188` | 300 |

와일드카드 레코드는 다음 서브도메인에 필요합니다:
- `alm.ske.myrobots.co.kr` — ArgoCD UI
- `monitoring.ske.myrobots.co.kr` — 모니터링 (활성화된 경우)
- `apps.ske.myrobots.co.kr` — UiPath Apps
- `insights.ske.myrobots.co.kr` — Insights (활성화된 경우)

### DNS 구성

DNS 제공자 (삼성 클라우드 DNS, 외부 DNS 또는 내부 DNS 서버)를 사용합니다:

```bash
# 예시: 삼성 클라우드 DNS 사용
# 네트워킹 > DNS > 호스팅 존 메뉴로 이동
# FQDN 및 와일드카드에 대한 A 레코드 추가

echo "DNS 구성:"
echo "  A 레코드: ${FQDN} → ${PUBLIC_IP}"
echo "  A 레코드: *.${FQDN} → ${PUBLIC_IP}"
```

## 3-7단계: DNS 확인 검증

```bash
# 기본 FQDN 확인
nslookup ${FQDN}
# 예상: Address: 123.41.32.188

# 와일드카드 확인
nslookup alm.${FQDN}
# 예상: Address: 123.41.32.188

# dig를 사용한 상세 확인
dig ${FQDN} +short
dig alm.${FQDN} +short
dig monitoring.${FQDN} +short
```

## 방화벽 규칙 요약

```
┌─────────────────────────────────────────────────────────────────┐
│                    방화벽 구성 요약                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. 인터넷 게이트웨이 방화벽 (IGW F/W):                          │
│     ┌─────────────────────────────────────────────────────┐     │
│     │ 인바운드: 출발지(전체) → LB IP:80,443  [허용]        │     │
│     └─────────────────────────────────────────────────────┘     │
│                                                                  │
│  2. 로드 밸런서 방화벽 (LB F/W):                                │
│     ┌─────────────────────────────────────────────────────┐     │
│     │ 아웃바운드: LB IP → 워커:30000-32767  [허용]         │     │
│     │ 인바운드: 헬스체크 → 워커:30000-32767  [허용]        │     │
│     │ 인바운드: SourceNAT → 워커:30000-32767  [허용]       │     │
│     │ 인바운드/아웃바운드: 80, 443  [허용]                  │     │
│     └─────────────────────────────────────────────────────┘     │
│                                                                  │
│  3. 보안 그룹 (워커 노드):                                      │
│     ┌─────────────────────────────────────────────────────┐     │
│     │ 인바운드: LB SourceNAT → NodePort 30000-32767 [허용] │     │
│     │ 인바운드: 헬스체크 → NodePort 30000-32767  [허용]    │     │
│     └─────────────────────────────────────────────────────┘     │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 검증 체크리스트

| 확인 항목 | 방법 | 예상 결과 |
|-----------|------|-----------|
| 퍼블릭 IP 할당 | 삼성 클라우드 콘솔 | IP 할당 완료 |
| IGW 방화벽 규칙 | 콘솔 > IGW > 방화벽 | 인바운드 80/443 허용 |
| LB 방화벽 규칙 | 콘솔 > LB > 방화벽 | 인/아웃바운드 NodePort 허용 |
| 보안 그룹 규칙 | 콘솔 > 보안 그룹 | LB에서 NodePort 허용 |
| DNS A 레코드 | `nslookup ${FQDN}` | 퍼블릭 IP로 해석 |
| DNS 와일드카드 | `nslookup alm.${FQDN}` | 퍼블릭 IP로 해석 |

## 문제 해결

### NLB가 응답하지 않는 경우

```bash
# NLB 프로비저닝 확인 (Istio 배포 후)
kubectl get svc istio-ingressgateway -n istio-system

# NodePort 할당 확인
kubectl get svc istio-ingressgateway -n istio-system -o jsonpath='{.spec.ports[*].nodePort}'

# 삼성 클라우드 콘솔에서 NLB 헬스 체크 상태 확인
```

### DNS가 해석되지 않는 경우

```bash
# DNS 전파 확인
dig ${FQDN} @8.8.8.8

# 내부 DNS 사용 시 네트워크 내에서 확인
dig ${FQDN} @<내부-dns-서버>

# 충돌하는 DNS 레코드 확인
dig ${FQDN} ANY
```

### 방화벽이 트래픽을 차단하는 경우

```bash
# 인터넷에서 NLB로의 연결 테스트
curl -vk https://${PUBLIC_IP} 2>&1 | grep -i "connect"

# 트래픽이 워커 노드에 도달하는지 확인 (워커 SSH 접속)
sudo tcpdump -i any port 32437 -n  # 32437 = HTTPS NodePort (예시)

# 삼성 클라우드 콘솔에서 방화벽 로그 확인
```
